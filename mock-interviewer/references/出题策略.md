## 一、简历锚点提取规则

读完简历后，从每个项目中提取以下类型的"可追问锚点"——那些写了但没展开的关键信息点：

### 技术选型锚点
简历中提到具体技术/工具/框架的地方：
- "用了X" → 为什么用X不用Y？X的原理是什么？遇到什么限制？
- "基于X实现" → X的核心机制理解多深？有没有踩坑？
- "接入X" → 接入过程中的技术难点是什么？

### 量化数据锚点
简历中出现的所有数字都是高价值追问点：
- "提升X%" → 怎么测的？AB实验还是前后对比？置信度多少？
- "日均X" → 峰值呢？低谷呢？波动大吗？
- "X人团队" → 怎么分工？怎么考核？怎么招的？

### 架构设计锚点
简历中描述系统/架构的地方：
- "搭建X系统" → 整体架构是什么？为什么这样设计？瓶颈在哪？
- "从0到1" → 调研了什么？最初的MVP是什么样？怎么演进的？
- "重构/升级" → 老架构什么问题？新架构怎么解决的？迁移怎么做的？

### 业务决策锚点
简历中涉及业务判断的地方：
- "主导X项目" → 为什么做这个？谁发起的？优先级怎么定的？
- "解决X问题" → 问题是怎么发现的？为什么是你来解决？
- "贡献X业务指标" → 归因怎么做的？有多少是自然增长？

### 团队管理锚点
简历中涉及团队/协作的地方：
- "带领X人团队" → 团队怎么组建的？怎么分工？遇到什么管理挑战？
- "协同X团队" → 跨团队协作的难点是什么？怎么推动的？
- "推广至全司" → 怎么推的？遇到什么阻力？怎么让别人买单？

## 二、JD 锚点提取规则（有 JD 时）

当用户提供了目标岗位的 JD，额外从 JD 中提取出题锚点。JD 锚点的优先级高于纯简历锚点——真实面试官手里拿着 JD，他们最关心的是"这个人能不能满足岗位要求"。

### 提取步骤

1. **列出 JD 核心要求**：从 JD 中提取 5-8 条关键能力/经验要求
2. **逐条匹配简历**：每条要求标注匹配度：
   - **强匹配** — 简历有直接对应的经历和数据
   - **弱匹配** — 简历有相关经历但深度/规模不够
   - **无匹配** — 简历没有体现
3. **聚焦 gap 出题**：弱匹配和无匹配的要求是面试重点——这些正是真实面试官会反复追问的地方

### JD 锚点类型

**硬性技能要求**
- JD 要求"分布式系统经验"但简历只有单体 → 追问："你有没有了解过分布式架构？如果让你把现有系统改成分布式，你会怎么做？"
- JD 要求"大规模数据处理"但简历数据量级不大 → 追问："你处理过的最大数据量是多少？如果涨100倍你的方案还能用吗？"

**软性能力要求**
- JD 要求"跨部门协调能力"→ 从简历中找跨团队合作经历追问，没有则用假设场景
- JD 要求"带领10人以上团队"但简历写带8人 → 追问团队管理的深度和挑战

**业务领域要求**
- JD 属于金融行业但候选人来自电商 → 追问行业理解和迁移能力
- JD 强调合规/安全但简历没提 → 用压力题探测："你的系统出了数据安全问题怎么办？"

### 出题优先级（有 JD 时）

```
JD gap 区域（必问）> JD 强匹配区域（验证深度）> 简历锚点（补充探测）
```

## 三、追问模式

### 漏斗式追问（P7技术面常用）
从表面往深处钻：
```
"你用了[某技术]"
  → "核心原理你怎么理解的？"
  → "实际使用中遇到过什么限制？"
  → "当时怎么解决的？有没有更好的办法？"
```

### 对比式追问（P8 Tech Lead常用）
用对比逼出深层理由：
```
"你选了方案A"
  → "对比过方案B吗？"
  → "A和B的核心差异在哪？"
  → "在什么条件下你会选B？"
  → "如果现在重选呢？"
```

### 推演式追问（P9架构师常用）
从当前往未来推演：
```
"你搭建了X系统"
  → "如果用户量涨10倍，哪里先扛不住？"
  → "你会怎么改？"
  → "改了之后下一个瓶颈在哪？"
```

### 质疑式追问（业务面常用）
质疑数据和归因：
```
"你说提升了X%"
  → "这个数据怎么算的？"
  → "排除外部因素后呢？"
  → "如果不做这个项目，这个数字会是多少？"
```

### 开放式追问（VP/CTO常用）
用开放问题探底：
```
"你怎么看[某方向]的未来？"
  → "具体说说？"
  → "你有什么依据？"
  → "如果你判断错了呢？Plan B是什么？"
```

## 四、压力题库

### 方案被否型
- "如果你精心设计的方案被上级否了，你怎么办？"
- "你和另一个团队的方案冲突了，怎么解决？"
- "产品经理说你的方案太复杂了，要求简化，你怎么取舍？"

### 故障场景型
- "这个系统现在挂了，线上影响多大？你的第一反应是什么？"
- "大促期间你负责的服务突然延迟飙升，你怎么排查？"
- "你负责的功能上线后出了严重bug，已经影响到用户了，你怎么处理？"

### 反转型
- "你说效率提升了X倍，如果没用这个方案你们会怎么做？"
- "这个项目如果不是你来做，换个人能做到一样的效果吗？"
- "如果你依赖的核心外部服务价格涨10倍，你的方案还成立吗？"

### 竞品对比型
- "竞品也在做类似的事，你觉得你们的壁垒是什么？"
- "如果行业头部公司做了一模一样的产品，你怎么赢？"
- "市面上有开源/现成的解决方案，你为什么还要自研？"

### 自我认知型
- "你觉得自己最大的技术短板是什么？"
- "你做过最失败的项目是什么？学到了什么？"
- "你的团队成员会怎么评价你？"

## 五、各面试类型的出题重点

### 项目深挖
- 锚点题为主，覆盖候选人最核心的2-3个项目
- 从项目背景问起，逐步深入到技术细节/业务决策
- 每个项目追问3-5个问题

### 技术原理
- 基于简历中提到的技术栈，考察底层理解
- 不是背概念，而是"你在实际使用中怎么理解这个技术"
- 结合具体场景提问："你用了[某技术]，在你的场景下[某方面]是怎么优化的？"

### 系统设计
- 基于简历项目变形："如果让你重新设计这个系统，面向更大规模/更多场景，你怎么设计？"
- 或者给一个相关但不同的场景
- 关注：需求分析→架构设计→核心模块→扩展性→取舍

### 行为面试
- 基于简历中的团队管理、跨团队协作经历
- 常见主题：冲突处理、优先级决策、失败经历、带人经验、向上管理
- 用STAR追问：情境→任务→行动→结果

## 六、教练模式引导策略

当候选人触发教练模式时，根据卡住的类型选择引导策略：

### 不知道从哪切入
**策略：定位问题**
> "你是完全没思路，还是有想法但不知道从哪说起？"
> "这个问题的核心在问什么？你觉得面试官想听到什么？"

### 知道方向但说不清
**策略：提供结构**
> "试试分层来说：先说结论，再说理由，最后说数据支撑"
> "你可以按时间线来组织：当时的背景→你做了什么→结果是什么"

### 技术细节记不清
**策略：类比迁移**
> "原理记不清没关系，你在实际项目中是怎么用的？从使用经验反推原理"
> "你在做其他项目时用过类似的技术吗？有什么共性？"

### 业务数据答不上来
**策略：拆解估算**
> "精确数字记不住，那数量级呢？是几万还是几十万？"
> "你当时是怎么做监控的？日常关注哪些核心指标？"

### 被压力题打懵
**策略：回归第一性原理**
> "别急着回答。先想想，这个问题的本质在考察什么？"
> "换个角度：如果你是面试官，你问这个问题想听到什么？"
